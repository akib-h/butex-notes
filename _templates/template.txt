## Step 0 — Verify against the live repo before writing anything
Do this before generating a single file, every time:
1. **Check for existing files in the target folder.** If any topics in this
   chapter already have files, list them with their real filenames and note
   their order. Do NOT rename or renumber existing files, even if their
   numbering doesn't match syllabus order — that breaks links and git
   history. Continue numbering from the next free integer for new files only.
   The chapter's README (Output 0) is where syllabus order gets shown
   correctly, regardless of what the filenames say.
2. **Open 1-2 existing files in this subject/course** (this chapter if any
   exist, otherwise a sibling chapter/course) and mirror their exact
   frontmatter schema, heading style, and tone. Don't reuse this prompt's own
   Output 2 structure if it conflicts with what's actually in the repo —
   actual practice overrides documented convention.
3. **Fetch the repo's real root `README.md` and `CONTRIBUTING.md`** if
   reachable, rather than relying on what they're assumed to say. If they
   document a convention (e.g. naming case) that contradicts what the actual
   files do, follow the actual files and flag the contradiction in one line —
   don't silently pick one.
4. **Check whether a course-level README exists** (one level above the
   chapter folder) if this course has multiple chapters/modules. If missing,
   note it — it's needed for any chapter README to cross-link properly
   (see Output 0b).
5. **Check the real root README.md** for whether this course already has an
   entry in the Course Index table / Detailed Course Contents. If not, that's
   Output 5, not optional.

## Role
Act as a contributor to the `[REPO_NAME]` GitHub repository. Produce rigorous,
multi-source-researched notes on [CHAPTER/UNIT] for [COURSE CODE], matching
this repo's *actual* structure and conventions as confirmed in Step 0.

## Research standard (non-negotiable)
- Verify every definition, law, formula, and derivation against 2–3 credible
  sources before writing (textbooks, university OCW, domain references —
  HyperPhysics/MIT OCW for physics, IUPAC/LibreTexts for chemistry, Wolfram
  MathWorld for math, official docs/papers for CS).
- Paraphrase fully — no copied source text.
- Flag source disagreements (convention, sign, scope) in one line.
- Flag any hand-note notation that diverges from the standard form.

## Output 0 — Chapter index file (`README.md`)
One per chapter, exact filename `README.md` (GitHub auto-renders it). Mirror
the structure of this subject's existing best chapter README if one exists
(header meta block, About, Official Syllabus quoted verbatim, Topic Index
table, Quick Formula/Key-terms reference, Core References, footer back-links).
**Topic Index table uses true syllabus order**, linking to whatever the real
filenames are — file numbering and syllabus position will not always match
for chapters with prior partial work, and that's fine.

## Output 0b — Course-level README (only if missing and course has >1 chapter)
Minimal index one level up: course info table, a row per
chapter/module linking to each chapter's README, back-link to repo root.

## Output 1 — One file per missing syllabus topic
- Continue numbering from the next free integer in the target folder; never
  renumber existing files.
- Naming: `NN_snake_case_topic.md` (lowercase, underscore-separated, zero-padded) —
  unless Step 0 found this repo genuinely uses a different live convention.
- Frontmatter: match exactly what Step 0 found in real files. Don't add
  fields (chapter, topic number, prerequisites) that existing files don't have.

## Output 2 — Per-topic content (organic structure, not a fixed mold)
Numbered `## 1., 2., 3...` sections shaped by the topic, but every file must
contain, somewhere, in whatever order reads naturally:
1. Definition/statement of the law, theorem, mechanism, or model.
2. **Core result**, adapted to DISCIPLINE TYPE:
   - Math → theorem + full proof (name the technique).
   - Physics/experimental science → law/principle + full derivation +
     experimental basis + limits of validity.
   - Chemistry → mechanism/equation with reasoning per step.
   - CS → algorithm/model + correctness argument + complexity.
   If a full proof/derivation is out of scope, give a labeled sketch + reference.
3. Worked examples — minimum 3, increasing difficulty, fully solved.
4. At least one comparison/distinction table or real-world application, where
   the topic naturally supports one.
5. Visual aid(s) — see Output 3.
6. Practice problems — 3-5, **plain numbered list, no hidden/spoiler
   answers** (only use spoiler blocks for collapsible reference tables, never
   to hide solutions, unless Step 0 found this repo doing otherwise).
7. References — numbered list, textbook chapter citations + 3-5 verified web
   links, matching how sibling files in this subject already cite sources.

## Output 3 — Visual assets (reality-checked for GitHub rendering)
Default to what the repo already uses for this subject (check Step 0):
- **Mermaid** and **ASCII-art code blocks** — no extra asset files, render
  natively, zero maintenance. Prefer these unless the repo already has an
  established image-asset pattern for this subject.
- **SVG/PNG** — only if this subject's existing files already use generated
  images, or the diagram genuinely can't work as Mermaid/ASCII (e.g. precise
  data plots). Save into whatever asset path the repo already uses.
- **.ipynb** — for computational/interactive content only; link, don't inline.
- Animated GIF — only for genuinely sequential/dynamic processes.
Avoid PSD/BMP/TIFF/raw `.drawio`/XLSM — none render inline on GitHub.
Every visual needs alt text.

## Output 4 — Changelog snippet
- Never rewrite `CHANGELOG.md`.
- Check whether an `[Unreleased]` section is already open; if so, merge new
  bullets into its existing `### Added`/`### Changed` lists rather than
  creating a duplicate heading.
- Match the exact bullet format already in use (e.g. `- **path** — desc:`
  with indented `` `file.md` — desc `` sub-bullets).
- Deliver as a standalone fenced code block, ready to paste in manually.

## Output 5 — Root README.md update (diff snippet, never a full rewrite)
If Step 0 found this course missing from the root README:
1. One new row for the Course Index table, matching its exact column format.
2. One new subsection under Detailed Course Contents, matching the pattern of
   an existing course's subsection (heading anchor, module/file links).
Deliver both as standalone snippets to paste in, not an edited full file.

## Output 6 — Commit message (per the repo's real CONTRIBUTING.md convention)
`type(scope): short imperative description` — confirm the exact types/scope
convention from the real CONTRIBUTING.md in Step 0 rather than assuming.

## Notation
- LaTeX: `$...$` inline, `$$...$$` block. `mhchem` extension for chemistry.
- SI units by default; state once per file if the course uses another convention.
