# Prompt: University Course Notes Generator — [REPO_NAME] repo

## Fill in before each use
- SUBJECT: [e.g. Physics, Math, Chemistry, CS, Textile Engineering]
- COURSE CODE: [e.g. PHY-103]
- CHAPTER/UNIT: [e.g. Thermodynamics]
- DISCIPLINE TYPE: [theoretical/math | experimental science | computational | applied/engineering]
  → this determines how "Core results" (Output 2, item 3) is framed


### Topic list (ordered, cleaned — this is the generation order)
1. [topic 1]
2. [topic 2]
...

### Suggested file structure
```text
[COURSE_CODE]/
└── [Chapter-Name]/
    ├── README.md
    ├── 01-[topic-1-slug].md
    ├── 02-[topic-2-slug].md
    ...
```
(`README.md` is the chapter index — GitHub auto-renders it as the folder's
landing content, so it must keep this exact filename, no number prefix.
slug = lowercase, hyphen-separated, matches topic list order exactly)

## Role
Act as a contributor to the `[REPO_NAME]` GitHub repository. Produce rigorous,
multi-source-researched notes on [CHAPTER/UNIT] for [COURSE CODE], matching
this repo's existing structure and conventions (YAML frontmatter, LaTeX math,
Mermaid diagrams, spoiler-format practice problems, changelog workflow).

## Inputs (attach what's available; ask before starting if #1 is missing)
1. **Syllabus / topic list** (required) — see above. Drives file order/count.
2. **Hand notes / lecture PDFs** (optional) — secondary check only, for what
   was actually taught (notation, scope, emphasis). Never the primary source —
   every definition/law/result still gets verified externally.
3. **README.md / CONTRIBUTING.md** (optional, repo-root level) — naming,
   folder layout, frontmatter schema. If missing, default to this repo's
   existing pattern in the relevant course folder and state explicitly which
   conventions were assumed.
4. **CHANGELOG.md** (optional, format reference only) — never rewritten, only
   appended to (see Output 4).

## Research standard (non-negotiable)
- Verify every definition, law, formula, and derivation against 2–3 credible
  sources before writing (standard textbooks, university OCW, domain-specific
  references e.g. HyperPhysics/MIT OCW for physics, IUPAC/LibreTexts for
  chemistry, Wolfram MathWorld for math, official docs/papers for CS).
- Paraphrase everything in your own words — no copying source text.
- If sources disagree on convention, sign convention, or scope, say so in one line.
- If hand notes use non-standard notation/units, flag the mismatch with the standard form.

## Output 0 — Chapter index file (`README.md`)
Generate this once per chapter, in addition to the per-topic files. Must be
named exactly `README.md` (no number prefix) so GitHub renders it as the
folder's landing page. Contents:
1. Chapter/unit title + course info table (code, title, credit, hours/week, total hours).
2. 3–5 sentence overview: what this chapter covers and why, in plain language.
3. Ordered topic list, each item linking to its corresponding file (this is the
   chapter's navigation index).
4. Prerequisites — earlier chapters/courses this chapter assumes.
5. Optional but encouraged: a Mermaid flowchart/mindmap showing how the topics
   build on one another (e.g. law → consequence → application chains).
6. One-line link back to the repo's root README/CONTRIBUTING for contribution norms.

## Output 1 — One file per syllabus topic
- File count and order = TOPIC LIST order, 1:1.
- Naming: per the Suggested file structure above; confirm against any existing
  files in the folder before naming the first one.
- Frontmatter: title, course code, chapter, topic number, date (today's date),
  tags, prerequisite links to earlier topic files in the same chapter.

## Output 2 — Per-topic content template (every file gets all of this)
1. **Overview** — 2–4 sentences: why this matters, where it sits in the chapter.
2. **Definitions & key terms** — numbered, formal, each with a plain-English gloss.
3. **Core results** — adapt to DISCIPLINE TYPE:
   - Math → theorem, formal proof (state technique: direct/induction/contradiction).
   - Physics/experimental science → law or principle, full derivation, the
     experimental basis/evidence behind it, and limits of validity.
   - Chemistry → mechanism/equation, with reasoning for each step.
   - CS → algorithm/model, correctness argument, complexity analysis.
   If a full derivation/proof is out of scope, give a labeled sketch + reference link.
4. **Worked examples** — minimum 3, increasing difficulty, full step-by-step work.
5. **Applications** — at least one real-world/engineering/industry use case.
6. **Visual aid(s)** — see Output 3; minimum one per file.
7. **Practice problems** — 3–5, solutions hidden in `<details>` spoiler blocks.
8. **References** — 3–5 links, each with a one-line note on what it adds.

## Output 3 — Visual assets (reality-checked for GitHub rendering)
Renders natively on GitHub — use these by default:
- **Mermaid** — process flows, classification trees, cycles, relationships.
- **SVG/PNG** — plots, vector diagrams, geometric/physical constructions
  (generate via Python/matplotlib or similar, save into the repo's asset path).
- **.ipynb** — for anything computational/interactive; link from the markdown
  file, don't inline raw output.
- Animated **GIF** — only for genuinely sequential/dynamic processes, not by default.

Does NOT render inline on GitHub — avoid unless converted first:
- PSD, BMP, TIFF, raw `.drawio`, XLSM. If draw.io is the best tool for a
  complex diagram, export to SVG/PNG, embed that, link the source separately.

Every visual needs alt text. Reuse openly-licensed diagrams (Wikimedia
Commons, OCW) when correctly licensed and clearly attributed; otherwise generate one.

## Output 4 — Changelog
- Never rewrite the existing file.
- Output ONLY the new entries, in a fenced code block, matching the existing
  changelog's header/format, ready to paste in manually.

## Notation
- Math/physics/chemistry equations: LaTeX, `$...$` inline / `$$...$$` block.
- Chemical equations: `mhchem` LaTeX extension where applicable.
- Units: SI by default unless the course/source uses another convention — state it once per file if so.
